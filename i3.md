# i3 Research Findings for `odin-play`

This document records concrete findings from researching [`i3/i3`](https://github.com/i3/i3) as a reference for this Odin terminal multiplexer.

The goal is not to port i3. The useful parts are behavior, data-model invariants, and edge cases around layout, focus, moving, closing, and commands.

Sources inspected from the i3 `next` branch:

- `include/data.h`
- `src/con.c`
- `src/tree.c`
- `src/move.c`
- `src/workspace.c`
- `src/render.c`
- `src/commands.c`
- `parser-specs/commands.spec`
- i3 user guide, hacking guide, and IPC docs

## High-Value Findings

### 1. i3 separates child order from focus order

In i3, every `Con` has separate lists:

- `nodes_head`: tiling child order.
- `floating_head`: floating child order.
- `focus_head`: focus history/order for children.

Relevant source:

- `include/data.h`: `nodes_head`, `floating_head`, `focus_head` on `Con`.
- `src/con.c`: `con_focus()`.
- i3 IPC docs: `focus` is serialized as child node IDs in focus order.

`con_focus()` moves the focused child to the head of its parent's `focus_head`, then recursively does the same up the ancestor chain. This means focus is represented as a path through the tree, not only as a single global focused id.

Current repo implication:

- `Workspace.focused_pane_id` is useful, but not enough long-term.
- `Node.focused_child_index` is enough for simple split layout, but i3's focus stack is better for:
  - restoring focus after close/move,
  - tabbed/stacked visible child selection,
  - workspace focus restoration,
  - `focus parent` / `focus child`,
  - JSON/debug tree output.

Suggested Odin adaptation:

- Keep child order in `children`.
- Add either:
  - `focused_child_index` plus a helper that updates ancestors, or
  - `focus_order: [dynamic]^Node` later if MRU focus behavior matters.
- For now, a simple ancestor update helper would be a good next step:
  - when a pane receives focus, walk up parents and set each parent's `focused_child_index` to the child on the focus path.

### 2. i3's `split` command mostly changes layout context

In i3, `split vertical|horizontal|toggle` calls `tree_split()`.

Relevant source:

- `src/commands.c`: `cmd_split()`.
- `src/tree.c`: `tree_split()`.

Important behavior:

- If the focused container is a workspace with 0 or 1 child, i3 changes the workspace orientation/layout.
- If the parent split container has only one child, i3 changes that parent's orientation instead of wrapping again.
- Otherwise, i3 wraps the focused container in a new split container of the requested orientation.
- i3's split command does not itself open a new terminal/window. It changes where the next window will be inserted.

Current repo implication:

- This repo's `split_focused_pane()` currently creates a new pane immediately, which is correct for a terminal multiplexer/tmux-style UX.
- Still, i3's wrapping rules are useful:
  - reuse an existing same-orientation parent,
  - avoid redundant one-child split containers,
  - wrap the focused pane only when needed.

Current implementation already partially matches this:

- If parent kind matches, it inserts the new pane into that parent.
- Otherwise it creates a new container around the focused pane.

Potential improvement:

- Insert the new pane adjacent to the focused pane, not always at the end of the parent child list.
- If a split container has only one child, prefer changing its orientation over creating a deeper redundant split.
- Update focus along the whole ancestor path after splitting.

#### Detailed split mode logic / UX

In i3, "split mode" is not a global modal state. It is a persistent layout property in the tree. A split command changes the current insertion context so that future windows open beside or below the focused container.

Relevant source/docs:

- i3 user guide: "Splitting containers", "Manipulating layout", "Orientation for new workspaces".
- `parser-specs/commands.spec`: `split v|h|t|vertical|horizontal|toggle`.
- `src/commands.c`: `cmd_split()`, `cmd_layout()`.
- `src/tree.c`: `tree_split()`, `tree_open_con()`.
- `src/con.c`: `con_attach()`, `con_set_layout()`, `con_toggle_layout()`.
- `src/workspace.c`: `_workspace_apply_default_orientation()`, `workspace_attach_to()`.

##### Terminology

```text
splith / split horizontal:
  children are arranged left-to-right.
  a newly opened child appears to the right of the focused child.

splitv / split vertical:
  children are arranged top-to-bottom.
  a newly opened child appears below the focused child.
```

This naming is easy to misread: `split horizontal` means the container is split along the horizontal axis, producing side-by-side panes.

##### Key commands in i3's default/example config

Typical i3 bindings are:

```i3
bindsym $mod+h split h
bindsym $mod+v split v
bindsym $mod+e layout toggle split
bindsym $mod+s layout stacking
bindsym $mod+w layout tabbed
```

Command syntax:

```text
split horizontal
split vertical
split toggle
split h
split v
split t

layout splith
layout splitv
layout toggle split
layout stacking
layout tabbed
```

##### `split` versus `layout`

These commands are different:

- `split h/v/t` changes the insertion context around the focused container. It is mainly about where the next window will open.
- `layout splith/splitv` changes the current container's layout immediately. If a leaf pane is focused, i3 applies this to the leaf's parent container.
- `layout toggle split` toggles the current container between `splith` and `splitv`, rearranging all children of that container.

Example:

```text
Before:
  H [ A B C ]    focus B

open D without split command:
  H [ A B D C ]

split v; open D:
  H [ A V [ B D ] C ]

layout splitv:
  V [ A B C ]
```

So `split v` localizes the next split around `B`; `layout splitv` changes the whole parent layout.

##### When split mode activates

A split orientation becomes active in these cases:

1. Workspace creation:
   - new workspaces get `splith` or `splitv` based on `default_orientation`;
   - default `auto` uses horizontal on wide outputs and vertical on tall/rotated outputs.

2. `split horizontal` / `split vertical`:
   - changes or creates a split container around the focused target.

3. `split toggle`:
   - toggles the requested split orientation according to the current parent/current container.

4. `layout splith` / `layout splitv` / `layout toggle split`:
   - changes the active layout of the current container.

5. Opening a new window/pane:
   - the new child is inserted into the focused insertion container, usually the focused pane's parent;
   - the insertion container's layout determines whether it appears right or below.

There is no temporary one-shot variable like `next_split_direction`. The tree itself is changed first; later insertion follows the tree.

##### Exact `split h/v` behavior

For `split horizontal` or `split vertical`, i3's `tree_split()` behaves roughly like this:

```text
split(focused, requested_orientation):
  if focused is workspace:
    if workspace has 0 or 1 child:
      set workspace layout to requested_orientation
      return
    else:
      encapsulate all workspace children in a new split container
      focused = that new container

  parent = focused.parent

  if parent is a split container with exactly one child:
    set parent layout to requested_orientation
    return

  create new split container with requested_orientation
  replace focused with new split container in parent
  attach focused as the only child of new split container
```

The important detail: if a focused pane is in a parent with multiple siblings, i3 wraps the focused pane in a new split container, even if the requested orientation matches the parent. This is how i3 makes the next window split inside the focused pane's old rectangle instead of joining the outer parent.

Example:

```text
Before:
  H [ A B C ]    focus B

split h; open D:
  H [ A H [ B D ] C ]

split v; open D:
  H [ A V [ B D ] C ]
```

Without the split command, opening `D` would produce:

```text
H [ A B D C ]
```

##### `split toggle` behavior

User-facing rule from the i3 guide:

- If the target split container has a single window, toggle that split container's orientation.
- Otherwise, make the current window a split container with the opposite orientation compared to the parent container.

Examples:

```text
Single pane A:
  split toggle; open B
  -> opposite/default orientation depending on current workspace layout

V [ A B ] focus A:
  split toggle; open C
  -> V [ H [ A C ] B ]

H [ A B ] focus A:
  split toggle; open C
  -> H [ V [ A C ] B ]
```

##### Opening/inserting new panes

When a new i3 window appears, `tree_open_con()` normally attaches it to the focused pane's parent. `con_attach()` inserts it after the currently focused tiling child in that parent.

For this repo, that means a separate `open pane` command should behave like:

```text
open_pane():
  parent = focused_pane.parent
  insert new pane after focused_pane in parent.children
  focus new pane
```

If the workspace only has one pane and no parent split yet, use the workspace's default orientation to create the first split container.

##### Recommended terminal-mux interpretation

This repo currently combines "split" and "open a pane" into one command. To match i3 UX, that combined command should behave like:

```text
split_and_open(direction):
  apply_i3_split_context(direction)  // may wrap focused pane in a new split container
  open_pane_in_focused_parent()      // insert new pane after focused pane
```

This differs from simply appending a new pane to the nearest same-orientation parent.

Important example for current implementation:

```text
Before:
  H [ A B C ]    focus B

Current simple implementation might produce:
  H [ A B C D ]

Closer i3-compatible behavior should produce:
  H [ A H [ B D ] C ]
```

So the current `split_focused_pane()` behavior should eventually change if the goal is i3-like split UX.

##### Layout commands for this repo

Keep these separate from pane creation:

```text
layout splith:
  change focused pane's parent container to Split_Horizontal

layout splitv:
  change focused pane's parent container to Split_Vertical

layout toggle split:
  if parent is Split_Horizontal: change to Split_Vertical
  if parent is Split_Vertical: change to Split_Horizontal
  if parent is Stacked/Tabbed: restore last split layout, or default to Split_Horizontal
```

This separation matters because i3 users expect:

- `split h/v` to affect where the next pane appears;
- `layout h/v` to rearrange the current container immediately.

### 3. Directional focus climbs until it finds a matching orientation

i3 directional focus is implemented through `tree_next()` / `get_tree_next()`.

Relevant source:

- `src/tree.c`: `get_tree_next()` and `tree_next()`.
- `src/commands.c`: `cmd_focus_direction()`.

Behavior summary:

1. Convert direction to orientation:
   - left/right => horizontal,
   - up/down => vertical.
2. Starting at the focused container, walk upward.
3. Find the first ancestor whose layout orientation matches the direction and has more than one child.
4. Pick the previous/next sibling in that ancestor.
5. If the target sibling is a branch, descend through its focus path to the actual focused leaf.
6. If no sibling exists, i3 may wrap focus or move to another workspace/output depending on config.

Current repo implication:

- Implementing focus should not only look at direct siblings.
- The useful minimal behavior is:
  - no wrapping initially,
  - no cross-workspace directional focus initially,
  - climb ancestors until the relevant split orientation is found,
  - select previous/next sibling,
  - descend through `focused_child_index` until reaching a pane.

This will feel much closer to i3 than naive sibling-only focus.

### 4. Pane navigation logic to match i3 UX

This is the concrete focus/navigation behavior this repo should implement for pane focus commands.

Relevant i3 source:

- `src/commands.c`: `cmd_focus_direction()` maps `focus left|right|up|down|next|prev` to `tree_next()`.
- `src/tree.c`: `get_tree_next()` implements upward search, edge wrapping, and workspace/output fallback.
- `src/con.c`: `con_descend_focused()` chooses the focused descendant of a target branch.
- `src/util.c`: direction helpers:
  - left/right => horizontal orientation,
  - up/down => vertical orientation,
  - left/up => previous sibling,
  - right/down => next sibling.

#### Direction mapping

```text
left  -> orientation horizontal, position previous
right -> orientation horizontal, position next
up    -> orientation vertical,   position previous
down  -> orientation vertical,   position next
```

For this repo:

```odin
Direction :: enum { Left, Right, Up, Down }

orientation_from_direction :: proc(direction: Direction) -> Node_Kind {
    switch direction {
    case .Left, .Right:
        return .Split_Horizontal
    case .Up, .Down:
        return .Split_Vertical
    }
}

is_previous_direction :: proc(direction: Direction) -> bool {
    return direction == .Left || direction == .Up
}
```

`Tabbed` should behave as horizontal for navigation. `Stacked` should behave as vertical. This matches i3's `con_orientation()` behavior where tabbed is horizontal and stacked is vertical.

#### Main `focus direction` algorithm

Starting from the focused pane:

1. Convert the requested direction to an orientation and previous/next position.
2. Start at the focused pane node.
3. Walk upward through parents until reaching the workspace root.
4. At each parent:
   - if the parent orientation does not match the requested orientation, keep walking upward;
   - if the parent has fewer than two children, keep walking upward;
   - otherwise, find the child on the focus path under that parent;
   - try to select the previous/next sibling of that child.
5. If a sibling exists, focus that sibling's focused descendant.
6. If no sibling exists, this is an edge case. Apply focus wrapping rules.

Important: i3 focus navigation does **not** choose the geometrically nearest edge pane inside the target branch. It chooses the target sibling branch, then descends through that branch's saved focus path.

Example:

```text
Horizontal split:
  A
  Vertical split:
    B
    C   <- last focused in this branch
```

If focus is on `A` and the user runs `focus right`, i3 focuses `C`, not necessarily `B`, because the right-hand branch's focused descendant is `C`.

#### Descending into a target branch

When navigation selects a container branch, i3 calls `con_descend_focused()`.

For this repo, the equivalent is:

```text
descend_focused(node):
  while node is not Pane and node has children:
    index = node.focused_child_index
    if index is invalid: index = 0
    node = node.children[index]
  return node
```

This means every container should remember its focused child. Updating this path is required whenever a pane receives focus.

#### Updating focus path

When a pane becomes focused:

```text
focus_pane(workspace, pane):
  workspace.focused_pane_id = pane.id

  child = pane_node
  parent = child.parent
  while parent != nil:
    parent.focused_child_index = index_of(parent.children, child)
    child = parent
    parent = parent.parent
```

This mirrors i3's `con_focus()`, which moves the focused child to the head of every ancestor's focus list.

#### Edge behavior / wrapping

i3's default config is `focus_wrapping on`.

With `focus_wrapping on`:

- If the focused pane is at the edge of a matching split, remember the opposite-edge child as a wrap candidate.
- Continue walking upward looking for a larger split that allows movement in the requested direction.
- If a larger split has a valid sibling, focus that sibling's focused descendant.
- If no larger split or adjacent workspace/output is found, focus the remembered wrap candidate's focused descendant.

For this terminal mux, there are no outputs yet, so the practical default should be:

```text
At an edge, keep climbing. If no higher-level move is possible, wrap within the nearest matching split.
```

Examples:

```text
Horizontal: [A B C]
focus C, focus right -> A
focus A, focus left  -> C
```

```text
Horizontal:
  A
  Vertical:
    B
    C

focus C, focus right -> A
```

Explanation: `C` has no horizontal sibling inside its vertical parent, so search climbs to the horizontal parent. The vertical branch is already the rightmost child, so i3 wraps to `A` if there is no higher-level move.

```text
Horizontal:
  Vertical:
    A
    B
  C

focus B, focus right -> C
focus B, focus left  -> C  # wraps at the horizontal parent
```

This can feel non-geometric, but it is i3's tree-based UX.

#### Other i3 wrapping modes

These are useful if this repo later makes wrapping configurable:

- `focus_wrapping off`:
  - edge focus does not wrap;
  - search still climbs to higher ancestors;
  - if no higher move exists, focus stays unchanged.
- `focus_wrapping on`:
  - i3 default;
  - edge focus may move to another workspace/output first;
  - otherwise wraps within the nearest matching split.
- `focus_wrapping force`:
  - wrap immediately inside the matching parent;
  - do not climb out of that parent.
- `focus_wrapping workspace`:
  - wrap inside the workspace;
  - do not move to another workspace/output via directional focus.

Recommended for this repo: implement `focus_wrapping on` semantics first, without cross-workspace directional movement.

#### `focus next` / `focus prev`

i3 also supports `focus next` and `focus prev`.

Behavior:

- Inspect the focused node's parent orientation.
- If parent is horizontal/tabbed:
  - `next` means right,
  - `prev` means left.
- If parent is vertical/stacked:
  - `next` means down,
  - `prev` means up.

This is optional early. Direct `focus left|right|up|down` is more important.

#### `focus parent` / `focus child`

i3 can focus containers, not just windows:

- `focus parent` moves focus to the parent container, up to the workspace.
- `focus child` descends through the focus path.

For this repo, pane-only focus is simpler. If container focus is added later, it should be separate from terminal input focus so key input still goes only to a pane.

#### Pseudocode for this repo

```text
focus_direction(workspace, direction):
  focused = find_focused_node(workspace.root, workspace.focused_pane_id)
  if focused == nil:
    return false

  wanted_kind = orientation_from_direction(direction)
  previous = is_previous_direction(direction)
  node = focused
  wrap_candidate = nil

  while node.parent != nil:
    parent = node.parent

    if navigation_kind(parent.kind) == wanted_kind and len(parent.children) > 1:
      index = index_of(parent.children, node)
      if index < 0:
        return false

      next_index = index - 1 if previous else index + 1
      if next_index >= 0 and next_index < len(parent.children):
        target = descend_focused(parent.children[next_index])
        return focus_node(workspace, target)

      if wrap_candidate == nil:
        wrap_index = len(parent.children) - 1 if previous else 0
        wrap_candidate = parent.children[wrap_index]

    node = parent

  if wrap_candidate != nil:
    target = descend_focused(wrap_candidate)
    return focus_node(workspace, target)

  return false
```

`navigation_kind()` should map:

```text
Split_Horizontal -> Split_Horizontal
Tabbed           -> Split_Horizontal
Split_Vertical   -> Split_Vertical
Stacked          -> Split_Vertical
Pane             -> none
```

#### Difference between focus navigation and pane movement

Do not reuse i3's movement targeting rules for focus navigation.

- Focus navigation uses `con_descend_focused()`.
- Pane movement uses `con_descend_direction()` when moving into a neighboring branch.

That distinction matters:

- focusing into a branch restores the branch's last focused pane;
- moving into a branch targets the edge-most pane/container in the movement direction.

So for this repo:

- `focus left/right/up/down`: branch target = focused descendant.
- `move pane left/right/up/down`: branch target = edge descendant, later.

### 5. Resizing logic / UX

i3 resizing is also tree-based. It changes split percentages on the relevant parent container, rather than storing absolute pixel sizes on panes.

Relevant source/docs:

- i3 user guide: "Resizing containers/windows".
- `parser-specs/commands.spec`: `resize grow|shrink ...`, `resize set ...`.
- `src/commands.c`: `cmd_resize()`, `cmd_resize_tiling_direction()`, `cmd_resize_tiling_width_height()`, `cmd_resize_set()`.
- `src/resize.c`: `resize_find_tiling_participants()`, `resize_neighboring_cons()`.
- i3 default config: resize binding mode.

#### User-facing command syntax

```text
resize grow|shrink <direction> [<px> px [or <ppt> ppt]]
resize set [width] <width> [px | ppt]
resize set height <height> [px | ppt]
resize set [width] <width> [px | ppt] [height] <height> [px | ppt]
```

`direction` can be:

```text
left
right
up
down
width
height
```

Meanings:

- `left/right/up/down`: resize against the neighboring container in that direction.
- `width/height`: resize the focused container in that dimension, taking/giving space across the other siblings in the relevant split.
- `px`: pixel amount.
- `ppt`: percentage points.

In tiling mode, i3 prefers `ppt` when a command contains both `px or ppt`. The default i3 resize mode uses:

```i3
resize shrink width 10 px or 10 ppt
resize grow   width 10 px or 10 ppt
resize shrink height 10 px or 10 ppt
resize grow   height 10 px or 10 ppt
```

For tiling containers this is effectively a 10 percentage-point resize. For floating windows, pixel resizing is used.

#### Default resize mode UX

i3's default/example config uses a dedicated binding mode:

```i3
bindsym $mod+r mode "resize"

mode "resize" {
    bindsym $left  resize shrink width 10 px or 10 ppt
    bindsym $right resize grow   width 10 px or 10 ppt
    bindsym $up    resize shrink height 10 px or 10 ppt
    bindsym $down  resize grow   height 10 px or 10 ppt

    bindsym Return mode "default"
    bindsym Escape mode "default"
    bindsym $mod+r mode "default"
}
```

UX implication:

- Resize is normally a temporary keybinding mode.
- Enter with `$mod+r`.
- Use direction keys repeatedly.
- Exit with `Enter`, `Escape`, or `$mod+r` again.

For this terminal mux, this maps cleanly to:

```text
mod+r            enter resize mode
h / left         shrink width
l / right        grow width
k / up           shrink height
j / down         grow height
enter / escape   leave resize mode
```

#### Direction-specific resize

For `resize grow right`, i3 searches for two resize participants:

1. `first`: the focused container or an ancestor of it.
2. `second`: the neighboring sibling in the requested direction.

Search behavior from `resize_find_tiling_participants()`:

1. Convert direction to orientation:
   - left/right => horizontal,
   - up/down => vertical.
2. Start at the focused container.
3. Walk upward until a parent with matching orientation is found.
4. Ignore stacked/tabbed parents for resizing.
5. Pick the sibling in the requested direction:
   - left/up => previous sibling,
   - right/down => next sibling.
6. If no sibling exists at that level, keep walking upward.
7. If no pair is found before the workspace, resize fails.

There is no resize wrapping. Edge resize commands fail/no-op if no suitable neighbor can be found.

Examples:

```text
H [ A B C ] focus B
resize grow right  -> B grows, C shrinks
resize grow left   -> B grows, A shrinks
resize shrink left -> B shrinks, A grows
```

Nested example:

```text
H [ A V [ B C ] D ] focus C
resize grow down   -> C grows, B shrinks? no, C has no lower sibling, so search climbs; no vertical ancestor sibling below, so fails
resize grow up     -> C grows, B shrinks
resize grow right  -> V [B C] grows, D shrinks
resize grow left   -> V [B C] grows, A shrinks
```

Important: when the matching resize happens at an ancestor, the whole ancestor branch is resized. In the example above, `resize grow right` grows the entire `V [B C]` branch, not just `C`.

#### Width/height resize

`resize grow width` and `resize shrink width` are less directional.

For width:

- i3 searches horizontally.
- It first tries to resize against a left sibling.
- If there is no left sibling, it can use the right sibling.

For height:

- i3 searches vertically.
- It first tries to resize against a lower sibling.
- If there is no lower sibling, it can use an upper sibling.

Then i3 adjusts the current container's percentage and distributes the opposite change across the other siblings in that parent.

Example:

```text
H [ A B C ] focus B
resize grow width 10 ppt
```

Result:

```text
B gains 10 percentage points.
A and C each lose 5 percentage points.
```

For this repo, this is the most useful behavior for a resize mode because users expect arrow keys to grow/shrink the focused pane dimension regardless of which side has a neighbor.

#### Percentage-based model

i3 stores sizing as child percentages (`Con.percent`). Rendering converts percentages to pixels and fixes rounding so children exactly fill the parent.

Resize rules:

- A positive amount grows the first/current participant and shrinks the second/other participant.
- A negative amount shrinks the first/current participant and grows the second/other participant.
- Resizing is refused if either participant would become smaller than approximately 1 pixel.
- After changing percentages, i3 normalizes/fixes percentages for the parent.

Current repo implication:

- `Node.weights` are a good equivalent to i3's `Con.percent`.
- Resizing should mutate weights on the selected parent container.
- Bounds should remain derived from layout, not edited directly.

#### `resize set`

For tiling containers:

```text
resize set width 50 ppt
resize set height 30 ppt
resize set width 80 px height 20 px
```

Behavior:

- `ppt` sets the target child percentage in the relevant parent.
- `px` is converted to a percentage based on the relevant parent size.
- `0` means "do not resize this dimension".
- Width and height can be set independently.

For this repo, `resize set` can be deferred. Interactive grow/shrink is more important.

#### Recommended implementation for this repo

Implement resizing in this order:

1. Add `Resize_Direction` and `Resize_Mode` commands:

```text
Resize_Grow_Width
Resize_Shrink_Width
Resize_Grow_Height
Resize_Shrink_Height
```

2. Implement width/height resizing first:

```text
resize_dimension(workspace, dimension, delta_weight):
  focused = focused pane node
  orientation = Split_Horizontal if dimension == width else Split_Vertical

  node = focused
  while node.parent != nil:
    parent = node.parent
    if navigation_kind(parent.kind) == orientation and len(parent.children) > 1:
      index = index_of(parent.children, node)
      grow child[index] by delta_weight
      distribute -delta_weight across siblings
      refuse if any weight would fall below min_weight
      normalize weights
      return true
    node = parent

  return false
```

This matches i3's `width/height` resize UX better than strict edge resizing.

3. Add direction-specific resize later:

```text
resize_edge(direction, delta):
  find focused node or ancestor with a matching-orientation parent
  find sibling in requested direction
  grow/shrink focused branch against that sibling
  fail if no sibling exists before workspace root
```

4. Add resize key mode:

```text
mod+r      enter Resize mode
h/left     shrink width
l/right    grow width
k/up       shrink height
j/down     grow height
escape     return to Normal mode
enter      return to Normal mode
```

5. Use percentage/weight deltas, not fixed terminal-cell deltas, at first:

```text
resize_step = 0.10  // i3-like 10 percentage points
```

Later, support cell-based deltas if that feels better for terminal panes.

#### Edge cases to preserve

- Do not resize stacked/tabbed parents directly; resize the nearest split ancestor.
- If no matching split ancestor exists, no-op.
- Never allow a pane/branch to shrink below a minimum size.
- Keep weights repaired after add/remove/move/resize.
- Resizing a nested pane may resize an ancestor branch if that is the nearest place where the requested dimension exists.

### 6. Move / reorder pane UX

i3 pane movement is tree-based and more structural than focus navigation. It reorders the focused container within the layout tree, or moves it into/out of neighboring branches.

Relevant source/docs:

- i3 user guide: "Moving containers" and "Moving containers to workspaces".
- `parser-specs/commands.spec`: `move <left|right|up|down>`, `move container to workspace ...`.
- `src/commands.c`: `cmd_move_direction()`, `cmd_move_con_to_workspace_*()`.
- `src/move.c`: `tree_move()`, `insert_con_into()`, `attach_to_workspace()`.
- `src/con.c`: `con_move_to_workspace()`, `con_next_focused()`.
- `src/tree.c`: `tree_flatten()`.

#### User-facing command syntax

```text
move left
move right
move up
move down

move container to workspace 1
move container to workspace number 1
move container to workspace next
move container to workspace prev
move container to workspace current
move container to workspace back_and_forth
```

For tiling containers, the optional `px` / `ppt` amount in `move left 20 px` is only relevant to floating windows. For this terminal mux, directional move can ignore amounts.

Typical i3 bindings:

```i3
bindsym $mod+Shift+h move left
bindsym $mod+Shift+j move down
bindsym $mod+Shift+k move up
bindsym $mod+Shift+l move right

bindsym $mod+Shift+1 move container to workspace number 1
bindsym $mod+Shift+2 move container to workspace number 2
```

Moving a container to another workspace does **not** switch to that workspace unless the user chains commands:

```i3
move container to workspace 3; workspace 3
```

#### Direction mapping

Movement uses the same direction-to-orientation mapping as focus:

```text
left  -> horizontal, previous
right -> horizontal, next
up    -> vertical,   previous
down  -> vertical,   next
```

But movement target selection is different from focus target selection:

- focus into a branch uses that branch's saved focused descendant;
- moving into a branch uses the edge-most descendant in the movement direction.

#### Simple sibling reorder

If the focused pane's parent has the requested orientation and there is a sibling in the requested direction, i3 first handles the local case.

If the adjacent sibling is a leaf, i3 swaps the two siblings.

Examples:

```text
H [ A B C ] focus B
move right -> H [ A C B ]
move left  -> H [ B A C ]
```

```text
V [ A B C ] focus B
move down -> V [ A C B ]
move up   -> V [ B A C ]
```

This is the first behavior this repo should implement.

#### Moving into an adjacent branch

If the adjacent sibling is itself a container, i3 moves the focused container into that branch rather than swapping with the whole branch.

The target inside the branch is chosen by `con_descend_direction()`:

- moving left targets the rightmost descendant of the branch;
- moving right targets the leftmost descendant;
- moving up targets the bottommost descendant;
- moving down targets the topmost descendant;
- when the branch orientation does not match the movement direction, i3 follows the branch's focused child and keeps descending.

This is edge-based targeting, not focus restoration.

Example:

```text
H [ A V [ B C ] D ] focus A
move right
```

The adjacent sibling to the right is `V [ B C ]`, a branch. i3 moves `A` into that branch near the side closest to the movement boundary. The exact insertion point depends on that branch's orientation and focused path.

Practical repo rule:

```text
if adjacent sibling is a branch:
  target = descend_edge(adjacent_branch, direction)
  insert moved pane before/after target so it lands on the edge nearest the source
```

This can be added after simple sibling swaps.

#### Moving out of a nested split

If the focused pane cannot move within its direct parent, i3 climbs ancestors until it finds a parent with the requested orientation.

Example:

```text
H [ A V [ B C ] D ] focus C
move right
```

`C` has no horizontal sibling inside its direct vertical parent. i3 climbs to the outer horizontal split and moves `C` out of the vertical branch toward the right side of that branch.

Conceptual result:

```text
H [ A V [ B ] C D ]
```

The key UX is:

- move commands can pull a pane out of nested splits;
- movement is relative to the nearest ancestor where the requested direction makes sense;
- after movement, cleanup/flattening is needed to avoid stale one-child containers.

#### Edge behavior

Unlike focus navigation, move does not wrap around inside a split.

At an edge:

- i3 tries to move at a higher ancestor level.
- If there is an adjacent output in that direction, i3 may move the container to that output's visible workspace.
- If no output/workspace move is possible, the command is effectively a no-op.

For this terminal mux with no outputs:

```text
At an edge, climb ancestors. If no higher move is possible, no-op.
```

Do **not** wrap `move right` from the rightmost pane to the left side. That would differ from i3.

#### Orientation forcing

When no ancestor with the requested orientation exists, i3 may force the workspace orientation so movement can happen in that direction.

For this repo, keep this simpler:

- if no matching split ancestor exists, no-op initially;
- later, consider changing the workspace/root split orientation only if this feels important.

#### Move to workspace UX

i3's move-to-workspace behavior:

- `move container to workspace N` moves the focused container to workspace `N`.
- It creates the workspace if necessary.
- It does not switch to that workspace.
- The moved container is placed at the focused insertion point of the destination workspace.
- Focus on the source workspace moves to the next appropriate pane.
- If the user wants to follow the pane, they chain `workspace N` after the move.

For this repo:

```text
move_pane_to_workspace(id):
  remove focused pane/subtree from current workspace
  insert into target workspace near target workspace's focused pane
  keep current workspace active
  focus a remaining pane in current workspace
```

Optional follow behavior can be a separate command:

```text
move_pane_to_workspace_and_follow(id)
```

#### Focus after move

i3 preserves focus carefully:

- if the moved container remains on the current workspace, it stays focused;
- if moved to another workspace without following, focus on the source workspace falls back to another pane;
- destination workspace remembers the moved pane in its focus path.

For this repo:

- after directional move, keep the moved pane focused;
- after move-to-workspace without following, focus the nearest remaining pane in the source workspace;
- if following, switch to destination workspace and focus the moved pane.

#### Structural cleanup after move

i3 calls `tree_flatten()` after structural movement. This prevents redundant split chains from accumulating.

For this repo, always run cleanup after a move:

```text
cleanup_layout_tree(workspace.root):
  remove empty containers
  collapse one-child containers
  merge adjacent same-kind split containers if safe
  repair focused_child_index
  repair weights
```

#### Recommended implementation phases

1. Same-parent sibling swaps:

```text
H [ A B C ] focus B, move right -> H [ A C B ]
```

2. Move out to nearest matching ancestor:

```text
H [ A V [ B C ] D ] focus C, move right -> H [ A V [ B ] C D ]
```

3. Move into adjacent branches using edge descent.

4. Move pane to workspace `1..9`.

5. Add optional `move pane to workspace and follow`.

Useful helper ideas from i3:

- `lowest_common_ancestor(a, b)`.
- `child_containing_con_recursively(ancestor, con)`.
- `descend_edge(node, direction)` based on `con_descend_direction()`.
- `insert_node_near_target(node, target, before_or_after)`.
- `cleanup_layout_tree(root)` after structural moves.

### 7. Redundant container cleanup matters

i3 uses `tree_flatten()` to remove redundant nested split containers.

Relevant source:

- `src/tree.c`: `tree_flatten()`.

The documented case is a redundant chain like:

```text
workspace horizontal
  split vertical
    split horizontal
      child1
      child2
  child3
```

When split containers become redundant after moves/closes, i3 reattaches children to the appropriate parent and restores focus order.

Current repo implication:

- After closing or moving panes, add a cleanup pass.
- For this terminal mux, the cleanup can be simpler than i3:
  - remove empty containers,
  - if a split container has one child, replace it with the child,
  - if a split container has a child split container of the same kind, optionally merge them,
  - recalculate weights after cleanup.

This will prevent the layout tree from accumulating useless nesting.

### 8. i3 stores split size as child percentage

i3 has `Con.percent` on each child. Rendering uses that percent to allocate the parent's size.

Relevant source:

- `include/data.h`: `double percent`.
- `src/render.c`: `precalculate_sizes()`.
- `src/move.c`: `con_fix_percent()` calls after detach/attach.

Rendering behavior:

- If a child has `percent > 0`, use it.
- Otherwise, use equal division: `1.0 / child_count`.
- Round sizes to integers.
- Adjust rounding leftovers so children exactly cover the parent rectangle.

Current repo implication:

- Current `Node.weights` on the parent is fine and may be easier in Odin.
- But i3's approach suggests some rules:
  - missing/zero weights should fall back to equal split,
  - after add/remove/move, normalize or repair weights,
  - rendering should distribute rounding leftovers so child rectangles exactly fill the parent.

Current layout code already gives the last child the leftover space. That is simple and acceptable early.

### 9. Stacked and tabbed layouts use focus order for visibility

i3 renders all children of stacked/tabbed containers into the same content area, then raises/focuses the focused child. Non-focused children are hidden/unmapped by X behavior.

Relevant source:

- `src/render.c`: `render_con_stacked()` and `render_con_tabbed()`.
- `src/con.c`: hidden checks use the parent's `focus_head`.

Behavior:

- `stacked`:
  - reserve a decoration/title row per child,
  - visible content belongs to the focused child.
- `tabbed`:
  - reserve one tab bar row,
  - split the tab bar horizontally across children,
  - visible content belongs to the focused child.

Current repo implication:

- For terminal panes, stacked/tabbed should probably reserve terminal rows:
  - stacked: one row per child for labels,
  - tabbed: one row total for tabs.
- Only the focused descendant should receive the content rectangle.
- Hidden panes should not retain stale visible bounds. Consider adding a `visible` flag or clearing bounds for hidden panes during layout.

### 10. Workspace focus is just tree focus

i3 workspaces are containers in the same tree, and switching a workspace calls `workspace_show()` then focuses `con_descend_focused(workspace)`.

Relevant source:

- `src/workspace.c`: `workspace_show()`.
- `src/con.c`: `con_descend_focused()` / related focus helpers.

Behavior:

- A workspace remembers focus because its subtree has focus order.
- Switching workspaces descends through the workspace's focus path to restore the focused child.
- Empty old workspaces may be closed by i3; this repo's fixed `1..9` workspaces should not be deleted.

Current repo implication:

- Keep fixed numbered workspaces for now.
- Store focus per workspace as this repo already does.
- Eventually derive `focused_pane_id` from the tree focus path instead of maintaining it separately, or keep it as a cache updated by focus helpers.

### 11. Commands are a stable abstraction boundary

i3 commands are the public behavior layer. Keybindings and IPC execute command strings.

Relevant source:

- `parser-specs/commands.spec`.
- `src/commands.c`.

Useful syntax subset for this repo:

```text
split horizontal
split vertical
split toggle
focus left|right|up|down
focus parent
focus child
move left|right|up|down
workspace 1
workspace number 1
move container to workspace 1
layout splith|splitv|stacked|tabbed
fullscreen toggle
kill
```

Current repo implication:

- The existing `Command` enum is a good start.
- Later, add a string command parser that maps i3-like strings to the enum.
- Keep command parsing separate from key input parsing.
- This makes future config and IPC much easier.

### 12. IPC/tree dump would be useful for debugging

i3 exposes `GET_TREE` over IPC. The tree reply includes layout, child nodes, focused state, focus order, rectangles, workspace info, etc.

Relevant docs/source:

- i3 IPC docs: `GET_TREE`, `RUN_COMMAND`.
- `src/ipc.c`.

Current repo implication:

- A full IPC layer is not needed early.
- A debug `dump_tree()` command or JSON-ish tree print would be valuable soon.
- If a command parser is added later, a `RUN_COMMAND`-style local socket API could be built on top of it.

## Things Not Worth Pulling

Avoid copying or spending much time on:

- X11/EWMH/ICCCM code.
- RandR/output management.
- Dock areas.
- Floating windows and scratchpad behavior.
- i3bar/nagbar/tooling.
- Generated C parser machinery.
- X11 rendering details.
- Restart/reload machinery.

Those are mostly irrelevant for a terminal multiplexer.

## Concrete Recommendations for This Repo

### Near-term improvements

1. Add focus-path helpers:
   - `focus_pane(workspace, pane_id)`.
   - Update each ancestor's `focused_child_index`.
   - Keep `workspace.focused_pane_id` in sync.

2. Change split insertion behavior:
   - when splitting inside same-orientation parent, insert the new pane after the focused pane instead of appending to the end.

3. Add directional focus:
   - climb ancestors until orientation matches direction,
   - select adjacent sibling,
   - descend via focused child indexes,
   - implement i3 default edge behavior: keep climbing, then wrap within the nearest matching split if no higher move exists.

4. Add resize mode and width/height resizing:
   - `mod+r` enters resize mode,
   - left/right shrink/grow width,
   - up/down shrink/grow height,
   - mutate split weights, not pane bounds.

5. Add layout cleanup:
   - remove empty containers,
   - collapse one-child split containers,
   - repair weights.

6. Add simple tree dump output:
   - print child order,
   - print focused child index,
   - print pane ids and bounds.

### Medium-term improvements

1. Add pane movement in phases:
   - sibling swap first,
   - cross-container movement later,
   - branch insertion later.

2. Add tabbed/stacked rendering model:
   - reserve title/tab rows,
   - layout only the focused child as visible,
   - clear or mark hidden pane bounds.

3. Add direction-specific resize and `resize set` later:
   - `resize grow left/right/up/down`,
   - `resize set width/height`,
   - cell or percentage-point amounts.

4. Add command parser for an i3 subset:
   - start with split/focus/move/resize/workspace/layout/fullscreen/kill.

5. Consider replacing `focused_child_index` with focus history if close/move behavior starts to feel wrong.

## Mapping i3 Ideas to Current Odin Types

Current:

```odin
Node :: struct {
    kind: Node_Kind,
    parent: ^Node,
    pane: ^Pane,
    children: [dynamic]^Node,
    focused_child_index: int,
    weights: [dynamic]f32,
}
```

Suggested incremental evolution:

```odin
Node :: struct {
    kind: Node_Kind,
    parent: ^Node,

    pane: ^Pane,

    children: [dynamic]^Node,
    focused_child_index: int,

    // Existing parent-owned weights are fine.
    weights: [dynamic]f32,

    // Add later only if MRU focus history is needed.
    // focus_order: [dynamic]^Node,
}
```

Do not add `focus_order` until there is a concrete need. First add focus-path helpers around `focused_child_index`.
