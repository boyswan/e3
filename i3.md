# i3 Research Findings for `odin-play`

This document records concrete findings from researching [`i3/i3`](https://github.com/i3/i3) as a reference for this Odin terminal multiplexer.

The goal is not to port i3. The useful parts are behavior, data-model invariants, and edge cases around layout, focus, moving, closing, and commands.

Sources inspected from the i3 `next` branch:

- `include/data.h`
- `src/con.c`
- `src/tree.c`
- `src/move.c`
- `src/workspace.c`
- `src/render.c`
- `src/commands.c`
- `parser-specs/commands.spec`
- i3 user guide, hacking guide, and IPC docs

## High-Value Findings

### 1. i3 separates child order from focus order

In i3, every `Con` has separate lists:

- `nodes_head`: tiling child order.
- `floating_head`: floating child order.
- `focus_head`: focus history/order for children.

Relevant source:

- `include/data.h`: `nodes_head`, `floating_head`, `focus_head` on `Con`.
- `src/con.c`: `con_focus()`.
- i3 IPC docs: `focus` is serialized as child node IDs in focus order.

`con_focus()` moves the focused child to the head of its parent's `focus_head`, then recursively does the same up the ancestor chain. This means focus is represented as a path through the tree, not only as a single global focused id.

Current repo implication:

- `Workspace.focused_pane_id` is useful, but not enough long-term.
- `Node.focused_child_index` is enough for simple split layout, but i3's focus stack is better for:
  - restoring focus after close/move,
  - tabbed/stacked visible child selection,
  - workspace focus restoration,
  - `focus parent` / `focus child`,
  - JSON/debug tree output.

Suggested Odin adaptation:

- Keep child order in `children`.
- Add either:
  - `focused_child_index` plus a helper that updates ancestors, or
  - `focus_order: [dynamic]^Node` later if MRU focus behavior matters.
- For now, a simple ancestor update helper would be a good next step:
  - when a pane receives focus, walk up parents and set each parent's `focused_child_index` to the child on the focus path.

### 2. i3's `split` command mostly changes layout context

In i3, `split vertical|horizontal|toggle` calls `tree_split()`.

Relevant source:

- `src/commands.c`: `cmd_split()`.
- `src/tree.c`: `tree_split()`.

Important behavior:

- If the focused container is a workspace with 0 or 1 child, i3 changes the workspace orientation/layout.
- If the parent split container has only one child, i3 changes that parent's orientation instead of wrapping again.
- Otherwise, i3 wraps the focused container in a new split container of the requested orientation.
- i3's split command does not itself open a new terminal/window. It changes where the next window will be inserted.

Current repo implication:

- This repo's `split_focused_pane()` currently creates a new pane immediately, which is correct for a terminal multiplexer/tmux-style UX.
- Still, i3's wrapping rules are useful:
  - reuse an existing same-orientation parent,
  - avoid redundant one-child split containers,
  - wrap the focused pane only when needed.

Current implementation already partially matches this:

- If parent kind matches, it inserts the new pane into that parent.
- Otherwise it creates a new container around the focused pane.

Potential improvement:

- Insert the new pane adjacent to the focused pane, not always at the end of the parent child list.
- If a split container has only one child, prefer changing its orientation over creating a deeper redundant split.
- Update focus along the whole ancestor path after splitting.

### 3. Directional focus climbs until it finds a matching orientation

i3 directional focus is implemented through `tree_next()` / `get_tree_next()`.

Relevant source:

- `src/tree.c`: `get_tree_next()` and `tree_next()`.
- `src/commands.c`: `cmd_focus_direction()`.

Behavior summary:

1. Convert direction to orientation:
   - left/right => horizontal,
   - up/down => vertical.
2. Starting at the focused container, walk upward.
3. Find the first ancestor whose layout orientation matches the direction and has more than one child.
4. Pick the previous/next sibling in that ancestor.
5. If the target sibling is a branch, descend through its focus path to the actual focused leaf.
6. If no sibling exists, i3 may wrap focus or move to another workspace/output depending on config.

Current repo implication:

- Implementing focus should not only look at direct siblings.
- The useful minimal behavior is:
  - no wrapping initially,
  - no cross-workspace directional focus initially,
  - climb ancestors until the relevant split orientation is found,
  - select previous/next sibling,
  - descend through `focused_child_index` until reaching a pane.

This will feel much closer to i3 than naive sibling-only focus.

### 4. Pane navigation logic to match i3 UX

This is the concrete focus/navigation behavior this repo should implement for pane focus commands.

Relevant i3 source:

- `src/commands.c`: `cmd_focus_direction()` maps `focus left|right|up|down|next|prev` to `tree_next()`.
- `src/tree.c`: `get_tree_next()` implements upward search, edge wrapping, and workspace/output fallback.
- `src/con.c`: `con_descend_focused()` chooses the focused descendant of a target branch.
- `src/util.c`: direction helpers:
  - left/right => horizontal orientation,
  - up/down => vertical orientation,
  - left/up => previous sibling,
  - right/down => next sibling.

#### Direction mapping

```text
left  -> orientation horizontal, position previous
right -> orientation horizontal, position next
up    -> orientation vertical,   position previous
down  -> orientation vertical,   position next
```

For this repo:

```odin
Direction :: enum { Left, Right, Up, Down }

orientation_from_direction :: proc(direction: Direction) -> Node_Kind {
    switch direction {
    case .Left, .Right:
        return .Split_Horizontal
    case .Up, .Down:
        return .Split_Vertical
    }
}

is_previous_direction :: proc(direction: Direction) -> bool {
    return direction == .Left || direction == .Up
}
```

`Tabbed` should behave as horizontal for navigation. `Stacked` should behave as vertical. This matches i3's `con_orientation()` behavior where tabbed is horizontal and stacked is vertical.

#### Main `focus direction` algorithm

Starting from the focused pane:

1. Convert the requested direction to an orientation and previous/next position.
2. Start at the focused pane node.
3. Walk upward through parents until reaching the workspace root.
4. At each parent:
   - if the parent orientation does not match the requested orientation, keep walking upward;
   - if the parent has fewer than two children, keep walking upward;
   - otherwise, find the child on the focus path under that parent;
   - try to select the previous/next sibling of that child.
5. If a sibling exists, focus that sibling's focused descendant.
6. If no sibling exists, this is an edge case. Apply focus wrapping rules.

Important: i3 focus navigation does **not** choose the geometrically nearest edge pane inside the target branch. It chooses the target sibling branch, then descends through that branch's saved focus path.

Example:

```text
Horizontal split:
  A
  Vertical split:
    B
    C   <- last focused in this branch
```

If focus is on `A` and the user runs `focus right`, i3 focuses `C`, not necessarily `B`, because the right-hand branch's focused descendant is `C`.

#### Descending into a target branch

When navigation selects a container branch, i3 calls `con_descend_focused()`.

For this repo, the equivalent is:

```text
descend_focused(node):
  while node is not Pane and node has children:
    index = node.focused_child_index
    if index is invalid: index = 0
    node = node.children[index]
  return node
```

This means every container should remember its focused child. Updating this path is required whenever a pane receives focus.

#### Updating focus path

When a pane becomes focused:

```text
focus_pane(workspace, pane):
  workspace.focused_pane_id = pane.id

  child = pane_node
  parent = child.parent
  while parent != nil:
    parent.focused_child_index = index_of(parent.children, child)
    child = parent
    parent = parent.parent
```

This mirrors i3's `con_focus()`, which moves the focused child to the head of every ancestor's focus list.

#### Edge behavior / wrapping

i3's default config is `focus_wrapping on`.

With `focus_wrapping on`:

- If the focused pane is at the edge of a matching split, remember the opposite-edge child as a wrap candidate.
- Continue walking upward looking for a larger split that allows movement in the requested direction.
- If a larger split has a valid sibling, focus that sibling's focused descendant.
- If no larger split or adjacent workspace/output is found, focus the remembered wrap candidate's focused descendant.

For this terminal mux, there are no outputs yet, so the practical default should be:

```text
At an edge, keep climbing. If no higher-level move is possible, wrap within the nearest matching split.
```

Examples:

```text
Horizontal: [A B C]
focus C, focus right -> A
focus A, focus left  -> C
```

```text
Horizontal:
  A
  Vertical:
    B
    C

focus C, focus right -> A
```

Explanation: `C` has no horizontal sibling inside its vertical parent, so search climbs to the horizontal parent. The vertical branch is already the rightmost child, so i3 wraps to `A` if there is no higher-level move.

```text
Horizontal:
  Vertical:
    A
    B
  C

focus B, focus right -> C
focus B, focus left  -> C  # wraps at the horizontal parent
```

This can feel non-geometric, but it is i3's tree-based UX.

#### Other i3 wrapping modes

These are useful if this repo later makes wrapping configurable:

- `focus_wrapping off`:
  - edge focus does not wrap;
  - search still climbs to higher ancestors;
  - if no higher move exists, focus stays unchanged.
- `focus_wrapping on`:
  - i3 default;
  - edge focus may move to another workspace/output first;
  - otherwise wraps within the nearest matching split.
- `focus_wrapping force`:
  - wrap immediately inside the matching parent;
  - do not climb out of that parent.
- `focus_wrapping workspace`:
  - wrap inside the workspace;
  - do not move to another workspace/output via directional focus.

Recommended for this repo: implement `focus_wrapping on` semantics first, without cross-workspace directional movement.

#### `focus next` / `focus prev`

i3 also supports `focus next` and `focus prev`.

Behavior:

- Inspect the focused node's parent orientation.
- If parent is horizontal/tabbed:
  - `next` means right,
  - `prev` means left.
- If parent is vertical/stacked:
  - `next` means down,
  - `prev` means up.

This is optional early. Direct `focus left|right|up|down` is more important.

#### `focus parent` / `focus child`

i3 can focus containers, not just windows:

- `focus parent` moves focus to the parent container, up to the workspace.
- `focus child` descends through the focus path.

For this repo, pane-only focus is simpler. If container focus is added later, it should be separate from terminal input focus so key input still goes only to a pane.

#### Pseudocode for this repo

```text
focus_direction(workspace, direction):
  focused = find_focused_node(workspace.root, workspace.focused_pane_id)
  if focused == nil:
    return false

  wanted_kind = orientation_from_direction(direction)
  previous = is_previous_direction(direction)
  node = focused
  wrap_candidate = nil

  while node.parent != nil:
    parent = node.parent

    if navigation_kind(parent.kind) == wanted_kind and len(parent.children) > 1:
      index = index_of(parent.children, node)
      if index < 0:
        return false

      next_index = index - 1 if previous else index + 1
      if next_index >= 0 and next_index < len(parent.children):
        target = descend_focused(parent.children[next_index])
        return focus_node(workspace, target)

      if wrap_candidate == nil:
        wrap_index = len(parent.children) - 1 if previous else 0
        wrap_candidate = parent.children[wrap_index]

    node = parent

  if wrap_candidate != nil:
    target = descend_focused(wrap_candidate)
    return focus_node(workspace, target)

  return false
```

`navigation_kind()` should map:

```text
Split_Horizontal -> Split_Horizontal
Tabbed           -> Split_Horizontal
Split_Vertical   -> Split_Vertical
Stacked          -> Split_Vertical
Pane             -> none
```

#### Difference between focus navigation and pane movement

Do not reuse i3's movement targeting rules for focus navigation.

- Focus navigation uses `con_descend_focused()`.
- Pane movement uses `con_descend_direction()` when moving into a neighboring branch.

That distinction matters:

- focusing into a branch restores the branch's last focused pane;
- moving into a branch targets the edge-most pane/container in the movement direction.

So for this repo:

- `focus left/right/up/down`: branch target = focused descendant.
- `move pane left/right/up/down`: branch target = edge descendant, later.

### 5. Moving panes is more complex than focusing panes

i3 moving is implemented in `src/move.c`, especially `tree_move()` and `insert_con_into()`.

Important behavior:

- If moving within a parent with matching orientation and an adjacent sibling exists:
  - if adjacent sibling is a leaf, swap them,
  - if adjacent sibling is a branch, move into the bordering branch.
- If there is no matching-orientation parent, i3 may force workspace orientation or move to another output.
- Moving preserves focus stack order carefully using lowest common ancestor logic.
- After moves, i3 calls `tree_flatten()` to remove redundant split-container chains.

Current repo implication:

- Do not try to fully port i3 movement at once.
- Recommended phases:
  1. Swap with adjacent sibling when parent orientation matches direction.
  2. Climb ancestors for cross-container moves.
  3. Descend into adjacent branches using their focus path.
  4. Collapse/flatten redundant containers after the move.

Useful helper ideas from i3:

- `lowest_common_ancestor(a, b)`.
- `child_containing_con_recursively(ancestor, con)`.
- `insert_con_into(con, target, before_or_after)`.
- `tree_flatten(root)` after structural moves.

### 6. Redundant container cleanup matters

i3 uses `tree_flatten()` to remove redundant nested split containers.

Relevant source:

- `src/tree.c`: `tree_flatten()`.

The documented case is a redundant chain like:

```text
workspace horizontal
  split vertical
    split horizontal
      child1
      child2
  child3
```

When split containers become redundant after moves/closes, i3 reattaches children to the appropriate parent and restores focus order.

Current repo implication:

- After closing or moving panes, add a cleanup pass.
- For this terminal mux, the cleanup can be simpler than i3:
  - remove empty containers,
  - if a split container has one child, replace it with the child,
  - if a split container has a child split container of the same kind, optionally merge them,
  - recalculate weights after cleanup.

This will prevent the layout tree from accumulating useless nesting.

### 7. i3 stores split size as child percentage

i3 has `Con.percent` on each child. Rendering uses that percent to allocate the parent's size.

Relevant source:

- `include/data.h`: `double percent`.
- `src/render.c`: `precalculate_sizes()`.
- `src/move.c`: `con_fix_percent()` calls after detach/attach.

Rendering behavior:

- If a child has `percent > 0`, use it.
- Otherwise, use equal division: `1.0 / child_count`.
- Round sizes to integers.
- Adjust rounding leftovers so children exactly cover the parent rectangle.

Current repo implication:

- Current `Node.weights` on the parent is fine and may be easier in Odin.
- But i3's approach suggests some rules:
  - missing/zero weights should fall back to equal split,
  - after add/remove/move, normalize or repair weights,
  - rendering should distribute rounding leftovers so child rectangles exactly fill the parent.

Current layout code already gives the last child the leftover space. That is simple and acceptable early.

### 8. Stacked and tabbed layouts use focus order for visibility

i3 renders all children of stacked/tabbed containers into the same content area, then raises/focuses the focused child. Non-focused children are hidden/unmapped by X behavior.

Relevant source:

- `src/render.c`: `render_con_stacked()` and `render_con_tabbed()`.
- `src/con.c`: hidden checks use the parent's `focus_head`.

Behavior:

- `stacked`:
  - reserve a decoration/title row per child,
  - visible content belongs to the focused child.
- `tabbed`:
  - reserve one tab bar row,
  - split the tab bar horizontally across children,
  - visible content belongs to the focused child.

Current repo implication:

- For terminal panes, stacked/tabbed should probably reserve terminal rows:
  - stacked: one row per child for labels,
  - tabbed: one row total for tabs.
- Only the focused descendant should receive the content rectangle.
- Hidden panes should not retain stale visible bounds. Consider adding a `visible` flag or clearing bounds for hidden panes during layout.

### 9. Workspace focus is just tree focus

i3 workspaces are containers in the same tree, and switching a workspace calls `workspace_show()` then focuses `con_descend_focused(workspace)`.

Relevant source:

- `src/workspace.c`: `workspace_show()`.
- `src/con.c`: `con_descend_focused()` / related focus helpers.

Behavior:

- A workspace remembers focus because its subtree has focus order.
- Switching workspaces descends through the workspace's focus path to restore the focused child.
- Empty old workspaces may be closed by i3; this repo's fixed `1..9` workspaces should not be deleted.

Current repo implication:

- Keep fixed numbered workspaces for now.
- Store focus per workspace as this repo already does.
- Eventually derive `focused_pane_id` from the tree focus path instead of maintaining it separately, or keep it as a cache updated by focus helpers.

### 10. Commands are a stable abstraction boundary

i3 commands are the public behavior layer. Keybindings and IPC execute command strings.

Relevant source:

- `parser-specs/commands.spec`.
- `src/commands.c`.

Useful syntax subset for this repo:

```text
split horizontal
split vertical
split toggle
focus left|right|up|down
focus parent
focus child
move left|right|up|down
workspace 1
workspace number 1
move container to workspace 1
layout splith|splitv|stacked|tabbed
fullscreen toggle
kill
```

Current repo implication:

- The existing `Command` enum is a good start.
- Later, add a string command parser that maps i3-like strings to the enum.
- Keep command parsing separate from key input parsing.
- This makes future config and IPC much easier.

### 11. IPC/tree dump would be useful for debugging

i3 exposes `GET_TREE` over IPC. The tree reply includes layout, child nodes, focused state, focus order, rectangles, workspace info, etc.

Relevant docs/source:

- i3 IPC docs: `GET_TREE`, `RUN_COMMAND`.
- `src/ipc.c`.

Current repo implication:

- A full IPC layer is not needed early.
- A debug `dump_tree()` command or JSON-ish tree print would be valuable soon.
- If a command parser is added later, a `RUN_COMMAND`-style local socket API could be built on top of it.

## Things Not Worth Pulling

Avoid copying or spending much time on:

- X11/EWMH/ICCCM code.
- RandR/output management.
- Dock areas.
- Floating windows and scratchpad behavior.
- i3bar/nagbar/tooling.
- Generated C parser machinery.
- X11 rendering details.
- Restart/reload machinery.

Those are mostly irrelevant for a terminal multiplexer.

## Concrete Recommendations for This Repo

### Near-term improvements

1. Add focus-path helpers:
   - `focus_pane(workspace, pane_id)`.
   - Update each ancestor's `focused_child_index`.
   - Keep `workspace.focused_pane_id` in sync.

2. Change split insertion behavior:
   - when splitting inside same-orientation parent, insert the new pane after the focused pane instead of appending to the end.

3. Add directional focus:
   - climb ancestors until orientation matches direction,
   - select adjacent sibling,
   - descend via focused child indexes,
   - implement i3 default edge behavior: keep climbing, then wrap within the nearest matching split if no higher move exists.

4. Add layout cleanup:
   - remove empty containers,
   - collapse one-child split containers,
   - repair weights.

5. Add simple tree dump output:
   - print child order,
   - print focused child index,
   - print pane ids and bounds.

### Medium-term improvements

1. Add pane movement in phases:
   - sibling swap first,
   - cross-container movement later,
   - branch insertion later.

2. Add tabbed/stacked rendering model:
   - reserve title/tab rows,
   - layout only the focused child as visible,
   - clear or mark hidden pane bounds.

3. Add command parser for an i3 subset:
   - start with split/focus/move/workspace/layout/fullscreen/kill.

4. Consider replacing `focused_child_index` with focus history if close/move behavior starts to feel wrong.

## Mapping i3 Ideas to Current Odin Types

Current:

```odin
Node :: struct {
    kind: Node_Kind,
    parent: ^Node,
    pane: ^Pane,
    children: [dynamic]^Node,
    focused_child_index: int,
    weights: [dynamic]f32,
}
```

Suggested incremental evolution:

```odin
Node :: struct {
    kind: Node_Kind,
    parent: ^Node,

    pane: ^Pane,

    children: [dynamic]^Node,
    focused_child_index: int,

    // Existing parent-owned weights are fine.
    weights: [dynamic]f32,

    // Add later only if MRU focus history is needed.
    // focus_order: [dynamic]^Node,
}
```

Do not add `focus_order` until there is a concrete need. First add focus-path helpers around `focused_child_index`.
